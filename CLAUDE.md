# nopt-heart-up-ec-v2

SI Web Shoppingベースのデザインカスタマイズプロジェクト。
画面ごとに支給されるExcel設計資料を実装上の正とし、デザインガイドライン（`.doc`）に準拠した新規デザインHTMLを作成する。
元モック（`モック/`）は参照資料として利用するが、設計資料と差分があること自体は問題にしない。

## ビルドコマンド

- `npm run dev` — 開発（Gulp watch + BrowserSync localhost:3000）
- `npm run build` — ビルド（workbench → mock）

## 編集禁止ディレクトリ

- **`モック/` 配下のファイルは絶対に編集・削除・移動しない**（参照のみ）
- **`specs/` 配下のファイルは絶対に編集・削除・移動しない**（参照のみ）

## Excel設計資料

- 格納場所: `specs/`
- 画面ごとのExcel設計資料を `specs/` に格納する
- 実装時は該当画面のExcelを参照し、記載された要素・構成・遷移先を正とする

## 本案件における実装の基本方針

画面ごとに支給されるExcel設計資料（`specs/`）をもとにHTML/CSS/JSの実装を行う。
モックとの差分有無を過度に問題視しない。設計資料作成時点で、モックと異なる構成・遷移先についてはクライアントとパッケージ提供元の間で合意済みの前提である。

ただし、設計資料に明示されていない追加変更・挙動変更・構造変更は独断で行わない。
「モック差分は許容」と「自由に変更してよい」は同義ではない。

設計資料に記載された内容に違和感があっても、明確に破綻していない限りは「意図された仕様」として扱うこと。
改善提案は実装判断と分離して記録すること。

### 実装判断の優先順位

1. 画面ごとのExcel設計資料
2. 個別の制作指示・レビュー指摘・補足コメント
3. デザインガイドライン（`.doc`）
4. ベースモック（`モック/`）

### 独断で変更してはいけない事項

- 画面の追加・削除・統合・分割
- 画面遷移/画面フローの変更
- 表示要素の追加・削除、項目順の並び替え
- PC/Tablet/SP間での構造変更
- 共通パーツのHTML構造変更
- フォルダ構成・CSS読み込み順の変更
- 設計資料にない独自アニメーションやインタラクションの追加

### 配置調整に関する補足

- 設計資料およびガイドラインに反しない範囲で、表示要素の配置場所や余白調整は許容する
- ただし、機能および項目要素を欠落させないこと
- 配置調整ではなく、構造変更・フロー変更・表示要素の追加削除に当たる場合は独断で行わない

### 相談・確認が必要なケース

- Excel設計資料内の記載が矛盾している
- 状態差分の範囲が不明（モーダル、ドロップダウン、タブ等）
- PC/Tablet/SPで見せ方が変わるがルールが明記されていない
- 共通パーツかページ固有か判断が分かれる
- 設計資料どおりにするとガイドライン上の懸念が出る
- 動的処理で成立しづらいレイアウトになっている
- 同時表示が難しい要素があり別パターンを起こす必要がありそう
- 設計資料そのものに欠落の可能性がある
- 相対パス・ファイル命名・格納先のルールとぶつかる

## 納品フォルダ階層（ガイドライン3-6 厳密準拠）

mock/ のビルド出力はガイドライン3-6に完全準拠する。ディレクトリ構造はパッケージと同様の構造で提供する。親フォルダに新規フォルダを作成しない。

### ページ/パーツ

| 納品パス | 内容 | workbench ソース |
|---|---|---|
| `cms/pc/[category]/` | ページHTML | `workbench/pages/[category]/` |
| `cms/pc/parts/general/` | 汎用パーツ | `workbench/includes/parts/general/` |
| `cms/pc/parts/main/` | メインパーツ | `workbench/includes/parts/main/` |

### CSS/JavaScript/画像

> **【変更】CSS統合方針（クライアント指示）**
> クライアントからの指示により、パーツ個別CSSを廃止し、全スタイルを1ファイル（`design.css`）に統合する方針に変更。
> これに伴い `files/partsfiles/styles/` へのCSS出力は行わない。
> コンポーネントSCSSは `_` プレフィックス付きパーシャルとし、`main.scss` で `@use` して `design.css` に統合出力する。

| 納品パス | 内容 | workbench ソース |
|---|---|---|
| `files/commonfiles/styles/` | デザインCSS（design.css, reset.css） | `workbench/styles/main.scss` → design.css |
| `files/commonfiles/images/` | 共通画像 | モック/共通/ からコピー |
| ~~`files/partsfiles/styles/`~~ | ~~パーツ個別CSS~~ | ※ design.css に統合済み（出力なし） |
| `files/partsfiles/scripts/` | パーツ個別JS | `workbench/scripts/parts/` |
| `files/exfiles/` | 外部ライブラリ（jQuery, Bootstrap等） | モック/共通/ からコピー |
| `files/systemfiles/` | システムJS/CSS | モック/共通/ からコピー |
| `files/contentfiles/` | アクセス解析等 | モック/共通/ からコピー |

## ファイル構成ルール

- ページHTML: `workbench/pages/[category]/`
- 汎用パーツ: `workbench/includes/parts/general/`（header, footer等）
- メインパーツ: `workbench/includes/parts/main/[category]/`
- 共通SCSS: `workbench/styles/base/` → `design.css`（統合出力）
- パーツSCSS: `workbench/styles/components/`, `layout/`, `pages/` → `_` パーシャルとして `main.scss` に統合 → `design.css`
- 親フォルダに新規フォルダを作成しない

### ファイル命名規則（厳守）

**ページHTMLファイル名は日本語で命名する。英語名に変換しない。**

- 形式: `{日本語画面名}（{状態}）.html` または `{日本語画面名}.html`
- 状態違いは全角括弧 `（）` で付与する
- 既存の命名例:
  - `トップページ（ログイン前）.html`
  - `トップページ（ログイン後）.html`
  - `マイページ.html`
  - `マイページ（データなし）.html`
  - `商品一覧（画像形式）.html`
  - `商品詳細（単品）.html`
- **禁止例**: `catalog-list.html`, `product-detail.html`, `mypage.html`（英語名は使わない）

**featureブランチ名も日本語で命名する。**

- 形式: `feature/{日本語画面名}`
- 例: `feature/商品一覧`, `feature/マイページ`, `feature/商品詳細（単品）`
- **禁止例**: `feature/catalog-list`, `feature/product-detail`（英語名は使わない）

## 利用規格（ガイドライン2-1）

- DTD: HTML5
- CSS: CSS3
- Charset: UTF-8
- 改行コード: CR/LF または LF（混在させない）
- 使用ライブラリ: jQuery 3.7.1, Bootstrap 5.3.2

## CSSコーディングルール（ガイドライン3-3, 3-4）

1. **素タグへのスタイル指定禁止**: `td {}`, `a {}`, `h3 {}` 等は使わない。必ず `.class-name {}` で定義
2. **`:nth-child(n)` 禁止**: 順番変更に耐えられる設計にする
3. **BEM命名規則**: `.c-{block}__{element}--{modifier}`
4. **ページ固有CSSは個別ファイル**: `workbench/styles/pages/[name].scss` に作成
5. **CSS/JS/画像のファイル名はディレクトリ跨ぎでも一意**
6. **相対パスのみ使用**
7. **CSS読み込み順を変更しない**（head.htmlで定義済み）
8. **可読性の高い実装**: `outputStyle: 'expanded'`（非圧縮）
9. **rem単位を基本使用**: `rem()` 関数でpx→rem変換
10. **ベースパッケージのCSS不使用**: 元パッケージのcommon.css・各画面用CSS（login.css等）は使用せず、全て新規作成する

## HTMLコーディングルール（ガイドライン3-4）

1. **HTML5準拠**: `<!doctype html>`, 閉じタグ明示
2. **charset UTF-8**
3. **改行コード**: CR/LF または LF（混在させない）
4. **パーツHTML**: どの画面から呼び出されても同一のHTML構成
5. **レイアウト可変パターン**: 同一画面でも可変がある場合は別ページを作成

## JSコーディングルール（ガイドライン3-4, 3-6準拠）

- **共通のカスタムJS（main.js等）は作成しない**
- JSはパーツ単位で `workbench/scripts/parts/[パーツ名].js` に作成
- ビルドで `files/partsfiles/scripts/[パーツ名].js` に出力
- 表示制御がある場合は、初期表示と操作後状態の両方を確認前提で組む
- デザイン適用のための制御コードは可読性高く保つ
- モーダル、ドロップダウン、タブなどは状態変化を前提に確認する
- 独自スクリプトは必要最小限にする

## レスポンシブ（ガイドライン2-5, 2-6）

### フロント
- SP: 320px〜767px
- PC: 992px〜

### 店頭
- Tablet: 768px〜991px
- PC: 992px〜

- PCを基準に設計を理解し、Tablet/SPは崩れないように調整する
- 端末差分でレイアウト変更が必要でも構造変更は最小限とする
- 設計資料または明示指示がない限り、端末別に別UIを作らない
- Tablet/SPで挙動や構造を変えたくなった場合は確認対象とする

## タップ領域（ガイドライン3-1 + 本案件の社内実装基準）

- ガイドライン上の必須要件として、タップ可能要素（`<a>`, `<button>` 等）は正しく構築し、小さい要素でも可能な限り押下しやすさを考慮する
- 本案件の社内実装基準として、原則 **44×44px 以上**、可能であれば **48×48px 以上**を目安とする
- アイコン等、視覚上小さい要素でも余白（padding）を含めた押下領域で44×44px以上を満たすこと
- チェックボックス・ラジオボタンはラベル込みで44×44px以上を押下可能にする
- 隣接するタップ要素は **8px以上**、可能であれば12px程度の間隔を確保
- リスト項目のリンクは行全体をタップ領域にする（右端の矢印だけ押せる作りは避ける）
- hover前提でしか意味が伝わらないUIにしない

## パーツ設計ルール（ガイドライン3-3）

- ヘッダー、フッター、カテゴリツリー、メッセージは原則パーツ化
- パーツHTMLは`workbench/includes/parts/general/`に配置
- ページのメイン部品は`workbench/includes/parts/main/[category]/`に配置
- どの画面から呼び出しても共通の表示・挙動となるようにする
- 呼び出し元画面によってHTML構造を変えない
- class命名は部品単位とし、画面名を含めない（`c-header` ○ / `top-header` ×）
- 状態違いはmodifier classで吸収する（`c-message--error` 等）
- ページ側からパーツ内部への強い上書きは避ける
- 部品自身は内部余白を持ち、外側余白は呼び出し側が制御する
- id依存は避け、可能な限りCSS/JSはclass/data属性ベースで設計する
- ページ固有要素と共通パーツの責務を分離する

## 画面要素の実装ルール（ガイドライン3-5）

設計資料に記載された要素は原則として全て画面上に表現する。
見た目上不要に見える要素でも独断で削除しない。

同時に表示が難しい場合は、同一画面の複数パターンとして整理する。
1つのHTMLで無理に全状態を吸収しない。

商材や条件により表示要素が変化する部分は、変化パターンを明示し、パターンごとにHTMLを用意すること。

画面要素には以下も含む: 見出し、説明文、注意文、エラー文言、件数表示、空状態、リスト、テーブル、ページャー、ボタン、タブ、メッセージ表示、補足文言、条件によって表示されるブロック

### 状態差分チェックリスト

- データあり/なし、エラーあり/なし、メッセージあり/なし
- リスト件数少/多、ページャーあり/なし
- 長文/短文、画像あり/なし
- ボタン活性/非活性、選択済み/未選択
- 展開前/後、モーダル閉/開、ドロップダウン閉/開

## 動的処理を前提とした実装ルール（ガイドライン2-4）

固定件数・固定文字数・固定高さを前提にした実装は避ける。

- カード数、商品数、行数の増減
- タイトルや注釈文の長文化、数字の桁数増加
- バナーや画像の有無、エラー表示の有無
- リストの複数行展開、ページャー表示有無
- 動的な表示/非表示制御

動的処理で成立しづらい表現がある場合は確認事項として残す。

## コンポーネント作成手順

1. SCSSファイル: `workbench/styles/components/_[name].scss` に作成（`_` パーシャル）
2. `workbench/styles/main.scss` に `@use 'components/[name]';` を追加
3. HTMLパーツ: `workbench/includes/parts/` に作成
4. JSファイル（必要な場合）: `workbench/scripts/parts/[name].js` に作成 → `files/partsfiles/scripts/` に出力
5. ページHTMLで `@@include` を使用
6. `npm run dev` でビルド確認

## 実装開始前チェック

- 対象画面がどのExcel設計資料に対応するか
- 設計資料のどこまでが確定仕様か
- 状態差分が存在するか
- パーツ化すべき箇所があるか
- 動的要素があるか
- 長文/件数増減で崩れないか
- 端末幅差分で破綻しないか
- フォルダ格納先とファイル命名に問題がないか
- モックとの差分が「合意済み差分」か「未定義差分」か

## 納品前チェック

- 設計資料どおりに要素が揃っている
- 状態差分が必要に応じて用意されている
- フル桁/最小/データなしでも崩れない
- 各ブラウザ・各対象端末幅で初期表示確認をしている
- 表示切替やモーダル等の簡易動的挙動を確認している
- タップ領域が極端に小さくない
- 共通パーツの構造が画面ごとにぶれていない
- CSS/JS/画像の命名重複がない
- 相対パスで参照している
- フォルダ構成が指定どおりである
- 読み込み順を壊していない
- W3C Markup Validation Service等でHTML規格適合を確認している
- WEB上で各画面が確認できる環境を提供している（`npm run dev` → localhost:3000）

## レビュー指摘の横展開ルール（ガイドライン4-2）

初回レビュー指摘はその画面だけでなく、同種パーツ・同種レイアウト・同種画面にも横展開する。

横展開対象: 余白、タップ領域、class命名、CSS設計、パーツ構造、ファイル命名、フォルダ格納、状態差分の漏れ、可変要素への耐性、レスポンシブ時の崩れ

## 作業報告ルール

作業完了時は以下を報告すること。

- 対象画面/対象パーツ、参照した設計資料名
- 作成/更新したファイル、新規追加したパーツ
- 想定した状態差分
- モックとの差分のうち設計資料を正として採用した点
- 未確定事項/要確認事項
- 横展開が必要そうな懸念点

## 作業フローの停止ルール（厳守）

画面実装は以下のフェーズに分かれており、**各フェーズ完了後は必ず停止してユーザーの指示を待つ**。自動で次のフェーズに進まない。

1. **設計フェーズ**: parser → designer → 提案レポート → **★ 停止**
2. **実装フェーズ**: builder → 実装完了レポート → **★ 停止**
3. **デザイン調整**: ユーザーが見た目を確認 → 修正指示 → builder修正 → 再確認（ユーザーが「OK」と言うまでループ）
4. **検証・納品フェーズ**: checker → validator → deployer（**ユーザーが明示的に指示した場合のみ**）

**特に重要**: 実装完了（フェーズ2）後は、ユーザーが `npm run dev` でブラウザ確認し、見た目の作り込み・調整を行う時間を取る。checker/validator/deployer はユーザーが「検証して」「checkerから進めて」等と明示するまで絶対に実行しない。

## Git管理・納品フロー

### ブランチ構成

| ブランチ | 用途 | 内容 |
|---|---|---|
| `main` | ソース管理・安定版 | workbench/, 設定ファイル, mock/静的資材すべて |
| `feature/xxx` | 画面単位の開発作業 | mainから分岐、完了後削除 |
| `delivery` | 納品専用 | mock/の中身をルート直下に展開（ガイドライン3-6構成） |

### 納品コマンド

- `npm run deliver -- "メッセージ"` — ビルド → deliveryブランチにコミット
- `npm run deliver:zip` — deliver実行 → `dist/` にzip + 納品メモ同梱

### 納品シナリオ

- **Git共有可能**: deliveryブランチのみ共有（ソース非公開）
- **Git共有不可**: `npm run deliver:zip` でzip + delivery-note.txt を納品

### 作業フロー

1. `git checkout -b feature/xxx main` — 作業ブランチ作成
2. パーツ/ページ単位でコミット
3. `git checkout main && git merge feature/xxx` — mainにマージ
4. `npm run deliver -- "メッセージ"` — 納品
5. `git branch -d feature/xxx` — ブランチ削除
6. 先方確認 → OKなら次へ / NGなら `fix/xxx` ブランチで修正

### 納品スケジュール（ガイドライン4-3）

一括納品ではなく、完成した画面から順次納品する。初回納品で先方レビューを受け、指摘事項を横展開した状態で以降の画面を作成する。
